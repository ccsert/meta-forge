# 架构初始化：小红书 AI 草稿投递平台

## 1. 文档目标

本文档用于在 `project-definition` 基础上，明确首版系统边界、模块划分、关键数据流、接口边界与技术选型理由，作为后续任务拆解与业务仓库初始化的基础。

---

## 2. 架构目标

首版架构优先满足以下目标：

- 以轻量化技术栈快速搭建可交付 MVP
- 优先验证“小红书图文自动写入草稿”主链路
- 支持多账号隔离执行与登录恢复
- 支持图片素材记录与尽量不重复分配
- 支持 AI 内容生成能力的可替换接入
- 支持任务队列、重试和日志审计，为后续扩展预留空间

---

## 3. 系统边界

### 3.1 系统内负责的能力

- 平台用户、运营用户与小红书账号的管理
- 图库素材的上传、标注、可用状态管理
- 图片使用记录、去重过滤和候选分配
- 基于账号画像与主题的 AI 图文生成
- 草稿任务创建、排队、执行、重试、状态回写
- 使用 Playwright 将图文写入小红书草稿箱
- 登录失效时生成二维码截图并通知指定用户
- 提供后台页面查看任务、账号、素材和异常情况

### 3.2 系统外依赖

- 小红书网页端创作页面
- 大模型服务供应商
- 对象存储服务
- 通知通道服务，如企业微信或飞书
- Redis 与 PostgreSQL 基础设施

### 3.3 当前不负责的能力

- 自动正式发布到小红书
- 跨平台内容发布
- 高级图像编辑流水线
- 复杂计费、结算与开放平台能力

## 5. 模块划分

### 5.1 Admin Web

职责：

- 提供账号管理、素材管理、任务查看、通知查看页面
- 提供账号画像、素材标签、任务发起等操作入口
- 提供登录二维码展示页和任务异常查看页

技术方向：

- `React 19`
- `Vite`
- `shadcn/ui`
- 路由与状态管理尽量保持轻量，首版避免过重方案

### 5.2 API Server

职责：

- 暴露前端所需 API
- 管理用户、账号、素材、任务、草稿、通知等领域对象
- 触发任务入队与状态变更
- 提供执行日志、截图元数据与任务状态查询接口

技术方向：

- `Hono`
- `TypeScript`
- 按领域拆分路由和 service，避免一开始拆成大量微服务

### 5.3 AI Content Orchestrator

职责：

- 统一封装 AI 内容生成流程
- 根据账号画像与主题生成文案提纲、候选标题、正文与标签
- 与素材分配模块协作，形成最终图文草稿

技术方向：

- 优先评估 `Mastra` 作为 AI 编排层
- 通过 adapter 隔离模型供应商、prompt 模板与业务流程

### 5.4 Asset Service

职责：

- 处理图片上传、元数据记录、标签管理
- 生成文件哈希、感知哈希
- 提供按账号、按用户、按主题的去重过滤能力

### 5.5 Task Scheduler

职责：

- 管理草稿任务状态机
- 将任务投递到对应队列
- 控制同账号串行执行
- 对失败任务进行重试或转人工

技术方向：

- `BullMQ` + `Redis`

### 5.6 Draft Delivery Worker

职责：

- 拉取待执行草稿任务
- 启动对应账号的 Playwright 持久化会话
- 检查登录、上传图片、填写图文并保存草稿
- 记录执行截图、错误信息与任务事件

### 5.7 Notification Service

职责：

- 在登录失效时发送二维码截图通知
- 统一管理企业微信 / 飞书等通知渠道
- 记录通知发送状态与失败原因

## 6. 关键数据流

### 6.1 草稿生成与投递主流程

1. 运营用户选择账号并发起草稿任务，或系统按策略创建任务
2. API 写入 `post_tasks`，状态为 `pending`
3. 调度器将任务放入草稿执行队列
4. AI 编排层生成候选文案结构
5. 素材模块根据主题与去重策略挑选图片
6. AI 编排层输出最终标题、正文、标签与图片顺序
7. Worker 启动 Playwright，检查账号登录态
8. 若未登录，截取二维码并创建 `login_events`
9. 通知服务将二维码发送给对应用户
10. 用户扫码成功后，Worker 恢复执行
11. Worker 上传图片并写入图文内容到小红书草稿箱
12. 保存成功后更新任务状态为 `success`
13. 系统写入图片使用记录与执行日志

### 6.2 登录恢复流程

1. Worker 访问创作页发现账号未登录
2. 截图二维码区域并上传对象存储
3. 写入登录事件，状态为 `waiting_scan`
4. 通知服务发送消息到对应用户
5. Worker 轮询登录状态或由任务恢复器重新调度
6. 登录成功后继续执行原草稿任务

### 6.3 图片去重流程

1. 从图库中筛选符合标签与主题的候选图片
2. 过滤该账号历史已使用图片
3. 过滤同一平台用户近期已使用图片
4. 对全局高频使用图片施加惩罚分
5. 对候选集进行评分、随机扰动与排序
6. 选出满足数量要求的图片组合
7. 草稿成功后记录 `media_usage_logs`

## 7. 领域模型草案

### 7.1 核心实体

- `users`
  - 平台用户
- `xhs_accounts`
  - 小红书账号
- `content_profiles`
  - 账号内容画像
- `media_assets`
  - 图片素材元数据
- `media_usage_logs`
  - 图片使用记录
- `post_tasks`
  - 草稿任务
- `post_drafts`
  - 生成后的图文内容快照
- `browser_sessions`
  - Playwright 会话与登录状态
- `login_events`
  - 登录二维码与扫码恢复事件
- `notification_logs`
  - 通知发送记录

### 7.2 关键关系

- 一个 `user` 可拥有多个 `xhs_account`
- 一个 `xhs_account` 对应一个主 `content_profile`
- 一个 `post_task` 属于一个 `xhs_account`
- 一个 `post_task` 生成一个 `post_draft` 快照
- 一个 `post_draft` 关联多张 `media_asset`
- 一次成功草稿执行会生成多条 `media_usage_log`
- 一个 `xhs_account` 可关联一个活跃 `browser_session`
- 一个待扫码登录过程对应一个 `login_event`

## 8. 接口边界草案

### 8.1 前后端接口

首版建议提供以下 API 分组：

- `POST /api/accounts`
- `GET /api/accounts`
- `POST /api/accounts/:id/profile`
- `POST /api/assets/upload`
- `GET /api/assets`
- `POST /api/tasks/drafts`
- `GET /api/tasks/drafts`
- `GET /api/tasks/drafts/:id`
- `POST /api/tasks/drafts/:id/retry`
- `GET /api/login-events/:id`
- `GET /api/notifications`

### 8.2 内部模块接口

建议在代码层形成这些边界：

- `account-service`
- `asset-service`
- `content-service`
- `draft-task-service`
- `playwright-runner`
- `notification-provider`

这些模块之间通过明确 DTO 和 service interface 交互，不让 Playwright 细节渗透到业务路由层。

## 9. 技术选型理由

### 9.1 为什么用 Hono

- 轻量，适合首版快速搭建 API 层
- 路由与中间件模型清晰，便于按领域拆分
- 与 `TypeScript` 结合自然，适合做一体化服务

### 9.2 为什么用 React 19 + Vite

- 首版后台以内部运营使用为主，优先开发效率与响应速度
- `Vite` 启动快、构建快，适合中后台项目
- `React 19` 生态成熟，便于与现有组件体系结合

### 9.3 为什么用 shadcn/ui

- 适合快速搭建管理后台
- 可控度高，不强绑定重型设计系统
- 更利于后续按业务场景做定制化页面

### 9.4 为什么评估 Mastra

- 适合承接多步骤 AI 流程编排
- 能将“提纲生成 → 选图 → 正文生成”形成结构化流程
- 便于后续替换模型和调优 prompt

### 9.5 为什么仍保留 BullMQ

- 多账号执行天然是异步任务场景
- 需要队列、重试、延迟恢复和串行控制
- 比手写内存任务调度更稳定，也更易观测

## 10. 首版部署建议

首版建议采用单体应用 + worker 的轻量部署方式：

- 一个 `web/api` 服务
- 一个 `worker` 服务
- 一个 `postgres`
- 一个 `redis`
- 一个对象存储

部署原则：

- 业务逻辑先保持单仓单体，避免过早拆微服务
- Playwright worker 与 API 可分开进程，避免互相阻塞
- 每个小红书账号使用独立持久化会话目录

## 11. 风险与缓解策略

### 11.1 页面结构变动风险

缓解：

- 选择器统一收敛到 Playwright adapter 层
- 关键节点保留截图和失败上下文
- 保存草稿流程单独编写可回归验证用例

### 11.2 登录态不稳定风险

缓解：

- 每账号独立 session storage
- 登录恢复流程显式建模为 `login_event`
- 对长时间未恢复任务设置超时与人工介入状态

### 11.3 素材重复率过高风险

缓解：

- 将账号级去重设为强约束
- 将用户级去重设为高优先级软约束
- 在素材不足时允许人工补图库，而不是过度依赖图片变形

### 11.4 AI 输出不稳定风险

缓解：

- 对 AI 输出定义固定结构 schema
- 对标题、正文、标签建立基础校验器
- 必要时保留人工预览与重新生成能力

---

## 12. 建议的首批任务方向

建议后续拆解任务时优先按以下顺序推进：

1. 搭建 `Hono` API 与基础项目骨架
2. 搭建 `React 19 + Vite + shadcn/ui` 管理后台骨架
3. 实现账号模型、素材模型、任务模型基础表结构
4. 实现单账号 Playwright 登录态维护与草稿保存验证
5. 实现图片上传、哈希记录与使用日志
6. 实现 AI 内容编排服务与草稿快照存储
7. 实现二维码通知链路与任务恢复
8. 实现多账号队列调度与失败重试

## 13. ADR 建议

下一步建议至少补两条 ADR：

- ADR-001：为什么首版采用 `Hono` 单体 + worker 架构
- ADR-002：为什么 Playwright 会话采用“每账号独立持久化目录”

---

## 14. 当前结论

该项目首版架构应采用“轻量单体 API + 独立异步 worker + 外部基础设施”的方案：

- 前端使用 `React 19 + Vite + shadcn/ui`
- 后端使用 `Hono`
- AI 编排层优先评估 `Mastra`
- 异步执行使用 `BullMQ + Redis`
- 数据持久化使用 `PostgreSQL`
- 浏览器执行使用 `Playwright`

先以最小系统验证单账号草稿投递链路，再逐步扩展到素材去重、AI 编排和多账号调度。
