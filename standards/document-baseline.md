# 项目文档基线标准

## 目标

确保项目在进入大规模任务执行前，已经形成足够稳定的参考文档，使后续任务不依赖临时对话记忆。

## 核心原则

- 项目文档是后续所有任务的长期上下文来源
- 文档必须优先于大规模开发任务创建
- agent 应尽量依赖文档而不是依赖“上一次聊天”
- 文档要能支持新开的 workspace 独立启动任务

---

## 最低文档基线

一个项目若要进入持续迭代，业务仓库中至少应具备：

### 1. 项目定义文档
- 目标
- 用户角色
- 核心功能
- 非目标范围
- 约束条件
- 里程碑

### 2. 架构初始化文档
- 系统边界
- 模块划分
- 关键数据流
- 接口边界
- 技术选型原因

### 3. 任务拆解文档
- Epic / Feature / Story / Task 树
- `parent_of`
- `depends_on`
- 任务验收标准
- repo_scope

### 4. ADR / 关键决策文档
- 已做出的关键技术/架构决策
- 决策原因
- 影响范围

### 5. 项目 manifest
- 项目元数据
- 仓库坐标
- VibeKanban ID
- 自动化策略
- 文档入口索引

---

## 扩展文档基线

对中大型项目，建议再补：

- 领域模型文档
- API 契约文档
- 数据模型文档
- 部署与环境说明
- 测试策略文档
- 集成边界文档

---

## 执行约束

- 若文档基线未完成，只允许创建“文档补齐型任务”
- 若执行性任务依赖的信息尚未文档化，必须先补文档或补 ADR
- 新工作空间启动时，应优先装配文档入口而不是整仓库全文
