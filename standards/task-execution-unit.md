# 任务执行单元标准

## 目标

定义什么样的任务才适合被创建为 VibeKanban 中的可执行任务，并进入单独 workspace session 执行。

## 核心原则

### 1. 一个执行性任务必须可在单 workspace 中闭环

所谓“闭环”是指：

- 任务目标明确
- 输入上下文完整
- 改动边界清晰
- 验收标准明确
- 完成后可以独立验证

如果一个任务必须依赖大量临时口头说明、跨多个不确定模块、或需要多个 agent 同时协作才能落地，那么它还不是合格的执行性任务。

### 2. 任务拆解优先保证可执行，不优先追求形式层级漂亮

任务拆解的目标不是把树画得复杂，而是让每个进入 VibeKanban 的开发任务都能被 agent 稳定执行。

### 3. 文档先行，任务后置

如果项目还没有形成足够稳定的文档基线，就不应过早创建大量执行性任务。

---

## 可执行任务的准入条件

一个任务要进入 VibeKanban 并允许启动 workspace session，至少满足：

- 有唯一、明确的目标
- 有清晰的 repo scope
- 有明确输入文档列表
- 有明确输出结果
- 有明确 acceptance criteria
- 有明确 DoD profile
- 已声明 `depends_on`
- 不依赖未沉淀的口头上下文

---

## 不合格任务的典型表现

以下任务不应直接创建为执行性任务：

- “先把系统大概搭起来”
- “把权限相关都做一下”
- “把前后端联调完”
- “先把架构梳理清楚再说”
- “先把这里重构一下，具体看情况”

这些更适合先变成：

- 文档任务
- 架构任务
- 拆解任务
- 调研任务

---

## 推荐任务字段

执行性任务建议至少包含：

- `task_id`
- `title`
- `task_type`
- `repo_scope`
- `doc_inputs`
- `acceptance_criteria`
- `expected_outputs`
- `depends_on`
- `merge_policy`
- `callback_policy`
- `executor`

---

## 验证问题

在创建执行性任务前，必须能回答：

- 这个任务是否能由一个 workspace 独立完成？
- 如果 agent 只拿到列出的文档，是否足够开始工作？
- 任务完成后，是否能通过测试或检查项独立验证？
- 如果出现冲突，是否知道该升级为 follow-up issue 还是自行解决？

只要上述任一问题答案是否定的，这个任务就应继续拆解或先补文档。
