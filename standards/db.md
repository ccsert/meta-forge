# 数据库设计规范

> 适用于所有由本元仓库管理的业务项目中的数据库设计。

---

## 通用原则

- 表名使用 snake_case 复数形式
- 字段名使用 snake_case
- 主键使用 `id`，推荐使用 UUID 或自增整数
- 每张表包含 `created_at` 和 `updated_at` 时间戳
- 软删除使用 `deleted_at` 字段（值为 null 表示未删除）
- 外键字段命名格式：`{关联表单数}_id`

---

## 字段类型约定

| 用途 | 推荐类型 | 说明 |
|------|----------|------|
| 主键 | `uuid` 或 `bigint` | UUID 更适合分布式场景 |
| 短文本 | `varchar(n)` | 明确长度限制 |
| 长文本 | `text` | 不限长度的文本内容 |
| 枚举 | `varchar` + 应用层校验 | 避免使用 DB 原生 enum |
| 布尔值 | `boolean` | 默认值显式声明 |
| 时间戳 | `timestamptz` | 始终使用带时区的时间戳 |
| JSON | `jsonb` | PostgreSQL 推荐 jsonb |
| 金额 | `numeric(precision, scale)` | 不使用浮点类型 |

---

## 索引规范

- 主键自动创建索引
- 外键字段必须创建索引
- 高频查询的过滤字段应创建索引
- 联合索引注意字段顺序（高选择性字段在前）
- 唯一约束通过唯一索引实现
- 避免过度索引，每张表索引数量控制在合理范围

---

## 迁移规范

- 使用 migration 文件管理所有 schema 变更
- 每个 migration 文件只做一件事
- migration 文件命名包含时间戳和描述
- 必须支持回滚（down migration）
- 生产环境禁止手动修改 schema

---

## 关系设计

| 关系类型 | 实现方式 |
|----------|----------|
| 一对一 | 外键 + 唯一约束 |
| 一对多 | 外键在"多"侧 |
| 多对多 | 中间表 |

---

## 安全规范

- 密码存储使用 bcrypt / argon2，不存储明文
- 敏感字段（Token、密钥）加密存储
- 日志中不打印敏感字段
- 查询使用参数化语句，防止 SQL 注入
- 定期清理过期数据

---

## 性能建议

- 避免 `SELECT *`，只查询需要的字段
- 分页查询使用 cursor-based 或 offset-based
- 大表查询注意 EXPLAIN 分析
- 热数据考虑缓存（Redis）
- 写入密集场景考虑批量操作
